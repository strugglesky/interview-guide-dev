按你这个平台的模块依赖关系，推荐用“先打地基，再把一条主业务链路跑通，再扩展”的顺序做，避免一上来就做最难的语音/多端联动。

**建议完成顺序（从 0 到 1）**
- **0. 基础工程与运行闭环**：Gradle 多模块可构建、`App.java` 可启动、`application.yml/.env`、PostgreSQL/Redis 连接、OpenAPI、统一 `Result<T>` + `GlobalExceptionHandler`、日志规范。
- **1. 通用基础能力（common/infrastructure）**：`ErrorCode/BusinessException`、`@RateLimit` + `RateLimitAspect` + Lua 脚本、`LlmProviderRegistry`、`StructuredOutputInvoker`、`RedisService`、文件存储（S3/RustFS）和解析（Tika）的基础接口。
- **2. KnowledgeBase（最适合先落地 AI 能力）**：文档上传 → 解析 → 入库 → 向量化（pgvector）→ RAG 查询/聊天。把 Redis Stream 的“异步任务管道”先跑通一条（向量化）。
- **3. Resume（复用文件与 AI）**：简历上传/解析/去重 → AI 评分。复用：Tika、S3、StructuredOutput、异步任务（简历分析管道）。
- **4. Interview（核心业务闭环）**：会话创建 → AI 出题 → 提交答案 → AI 评估 → 报告导出（iText）。这一步把第三条异步管道（面试评估）跑通，并把“平台价值”最核心路径做完。
- **5. InterviewSchedule（增强体验）**：日历/事件模型 → 邀请邮件/文本解析（AI）→ 生成日程。它依赖 AI 与基础持久化，但不强依赖面试/语音。
- **6. VoiceInterview（最后做）**：WebSocket 实时通话、ASR/TTS、多轮评估、状态机与容错。这是系统复杂度最高、外部依赖最多的一块，放在最后能最大化复用前面成熟的评估与会话体系。

**每个阶段的“完成标准”**
- 有最小 API 集合可用（Controller 只做路由 + 校验）。
- 数据能落库、异常能返回统一结构、限流能生效。
- 至少一条异步任务（Redis Stream）能生产/消费/重试/失败标记跑通。
- 前端至少能把该模块的核心流程走一遍（哪怕很简陋）。
